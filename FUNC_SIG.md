# Function Signature Type Reinterpretation Summary

This document is organized around how the source C/C++ signature types were interpreted in the Rust translations. The main signal here is the type-shape change itself: arrays becoming fixed arrays, slices, or `Vec`; C strings becoming `&str` or `String`; raw pointers becoming references or owned containers; and primitive widths, signedness, or precision changing.

## Comparison Rule

- Primitive types are considered preserved only when signedness, width, and floating-point precision match.
- C/C++ pointers are considered preserved only when translated to Rust raw pointers with the same pointee type.
- Pointer-to-reference conversions are reported as changes, even when they look semantically convenient.
- C/C++ array parameters are reported as changes when translated to Rust slices, fixed arrays, `Vec`, `String`, or `&str`.
- `C-array<T>` means a C/C++ array parameter such as `T arr[]`; it is intentionally distinct from a Rust fixed array such as `[T; 2]`.
- The comparison is signature-level only. It does not prove behavioral equivalence or memory-layout equivalence.

## What To Notice

- `C-array<i32>` is often not represented as a pointer-like value in Rust. The largest groups are fixed arrays such as `[i32; 2]` and `[f32; 2]`, followed by slices like `&mut [i32]` and `&mut [f32]`. For an LLVM/Rocq model, this means the Rust side often lowers through Rust aggregate/slice conventions rather than preserving the C ABI shape.
- Some integer arrays also change element type, especially `C-array<i32>` to `[f32; 2]`, `&mut [f32]`, or `Vec<f32>`. These are not mere container changes; arithmetic domains change too.
- C string-like inputs split into several Rust shapes: `char[]`/`char *` to `&str`, `String`, `Vec<char>`, `&[u8]`, or `&[char]`. These imply different IR patterns: fat pointers for slices/`str`, owned allocation for `String`/`Vec`, and Unicode scalar handling for Rust `char`.
- C++ `string` is mostly translated as owned Rust `String`. That preserves the high-level notion of a string but not the ABI, ownership, allocator behavior, or LLVM type shape.
- Raw struct/data pointers in `c_crown` often become references or ownership encodings such as `&Node`, `&mut Option<Box<Node>>`, `Option<Box<Node>>`, or `Rc<Node>`. These are the cases where pointer modeling choices matter most.
- Scalar drift is common enough to track explicitly: `i32 -> u32`, `i32 -> usize`, `i64 -> i32`, `i32 -> f32`, and `f64 -> f32` appear in translated signatures.
